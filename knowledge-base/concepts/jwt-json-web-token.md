## Definition
Jwt (Json Web Token) is a compact, URL-safe means of representing claims to be transferred between two parties. It extends oauth-2-0 for secure information exchange in a web environment and builds on the OAuth 2.0 framework for access token credentials. Json Web Token is widely used in modern web applications to transmit user information securely and authenticate requests between a client and a server.

## How it works
Json Web Token (JWT) operates by conveying claims between client and server within a self-contained token according to its specification. A JWT consists of three parts: Header, Payload, and Signature. The Header typically consists of two parts: the token type (`JWT`) and the hashing algorithm used (e.g., `HS256`).

The Payload contains the claims (assertions), which may belong to one or more distinct claim sets: Registered, Public, and Private claims. Registered claims provide interoperability and useful guidance for token consumers, while public claims are defined by parties other than the IETF standards but shared between multiple parties. Private claims are those not intended for interoperability and are defined by the security system in place. These claims can encode user identity, roles, permissions, and other information needed for authentication and authorization.

The Signature ensures the token's integrity and authenticity, generated by encoding the Header and Payload, and then applying the selected hashing algorithm to that encoded string. This cryptographic signature is typically secured with a secret (known as a Symmetric Secret) or private/public key pair (known as a Public/Private Key). 

Encryption mechanisms in JWT can optionally be implemented to obfuscate the payload from third-party inspection, although encryption in JWT is less common than signing.

On the client side, a JWT token is generated after a successful authentication process and will be stored in either localStorage or cookies within the browser. For subsequent requests to the server, the client must include the JWT in the Authorization header as `Bearer <token>`. The server validates the JWT by checking its signature using the same secret used to generate it, or by verifying the HMAC signature. From the verified token, the server extracts user information necessary for authorization decisions.

The token can also be stored on the client side in various storage formats depending on the requirement for persistence and security. Session storage and local storage are commonly used for session-based tokens. JWT tokens can be signed or encrypted according to the jwt-signing-vs-encryption.

## Variants
Json Web Token has two main variants: signed JWTs and encrypted JWTs. Signed JWTs rely on a shared secret or a public-private key pair for validating the authenticity and integrity of the token, whereas encrypted JWTs go a step further by ensuring the content itself remains confidential. Encryption in JWT provides an additional layer of security by obfuscating the payload data, making it inaccessible to unauthorized parties.

## Trade-offs
The use of JWT introduces several trade-offs and limitations, including the risk of the token being intercepted or misused if the transmission or storage mechanisms are compromised. Once a token is compromised, it can be used to authenticate as the user until it expires. Additionally, while JWTs offer simplicity in implementation, they can become bulky when a large amount of data is included in the token payload, potentially leading to performance degradation due to the increased size of each request and response.

Moreover, endpoint protection relies heavily on secure transmission and storage measures, since the token can be used to impersonate the user. This means they are ideal for stateless server authentication but may not fully mitigate all security risks without additional precautions such as CSRF protection and secure session management.
