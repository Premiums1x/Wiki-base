
# 《Claude Code + Obsidian + sage-wiki：本地LLM知识库探索》项目报告

## 一、项目概况

### 1.1 项目背景

sage-wiki 是一个由 Go 语言编写的 LLM 驱动 Graph-RAG 知识库编译器（https://github.com/xoai/sage-wiki）。它读取原始 Markdown 文档，通过三阶段 LLM 编译管线自动生成结构化 Wiki 知识库。本项目基于 sage-wiki 复现了计算机科学多维度知识库，以 9 份学习路线图为原始资料，编译生成 101 个核心概念和 46 条本体关系。

### 1.2 技术栈

| 组件 | 技术选型 |
|---|---|
| 编译器 | sage-wiki (Go)，Graph-RAG 三阶段流水线 |
| LLM API | 硅基流动 (SiliconFlow)，Qwen2.5-7B/14B |
| 嵌入模型 | BAAI/bge-m3，1024 维，8192 token 上下文 |
| 搜索 | SQLite + BM25(0.7) + 向量检索(0.3) |
| 编辑器 / MCP | Obsidian / Claude Code |

### 1.3 与 RAG、普通笔记的对比

| 维度 | 普通笔记 | 传统 RAG | 本项目 (Graph-RAG) |
|---|---|---|---|
| 组织方式 | 手动建目录、标签 | 向量库 + 检索 | 自动提取概念 + 本体关系图谱 |
| 链接语义 | 扁平 `[[链接]]`，无类型 | 无显式链接 | extends/implements/depends on 等强类型 |
| 更新方式 | 手动编辑 | 重新切片索引 | 增量编译，仅处理变更源 |
| 可解释性 | 依赖人 | 黑盒相似度 | 图谱可追溯到原始 source |

核心差异：传统 RAG 是"检索-生成"两段式，本项目是"先构建本体知识网，再在知识网上检索"。查询时上下文不仅来自向量相似文本块，还包括图谱中的直接邻居节点（通过关系扩展），以及源文档溯源链接，检索的语义深度和可解释性显著提升。

设计理由：源文件按 01-09 编号前缀确保排序与学习顺序一致（从 AI/ML 到 网络安全，形成从理论到应用的递进逻辑），同时将输出目录从默认的 `wiki/` 改为 `knowledge-base/`，使输入、输出、配置职责清晰分离。这项配置完全通过 `config.yaml` 实现，无需修改 Go 源码。

对应的 `config.yaml` 源目录配置：

```yaml
sources:
  - path: raw/01-ai-and-machine-learning
    type: auto
    watch: true
  - path: raw/02-data-science-and-bigdata
    type: auto
    watch: true
  # ... 共 9 个源目录
output: knowledge-base
```

## 三、特征工程

### 3.1 编译管线

sage-wiki 的核心特征工程是**三阶段 LLM 编译管线**：

**阶段一：Summarize（摘要）** — LLM 读取每个源文档，按 Key claims / Methodology / Results / Concepts 结构生成结构化摘要，存入 `summaries/`。

**阶段二：Extract（概念提取）** — LLM 分析所有摘要，提取命名实体。每个实体包含 name（kebab-case 标识符）、aliases（别名）、sources（来源）、type（concept / technique / claim）。这里的关键是实体类型分类——详见第六节的 AI 协作排错。

使用的 Prompt 模板（`extract_concepts.txt`）中包含类型分类指南：

```text
**technique** — 方法、算法、流程、实践。问："这是做某事的方式吗？" 是 → technique
  示例：supervised-learning, self-attention, pre-training, RAG, flexbox
**concept** — 工具、框架、语言、协议、领域。问："这是一个事物/工具/领域吗？" 是 → concept
  示例：python, react, kubernetes, http, docker
**claim** — 源自源文档的具体断言或发现。问："这是一个具体陈述吗？" 是 → claim
  示例："attention is all you need", "pretraining improves convergence"
```

**阶段三：Write（文章写作）** — LLM 为每个概念撰写结构化 Wiki 文章，包含 Definition / How it works / Variants / Trade-offs / See also 五个标准章节。自定义 Prompt（`prompts/write-article.md`）中强制要求使用关系动词搭配 `[[wikilink]]`：

```text
CRITICAL: 在正文中使用关系词搭配双括号链接：
- "extends", "builds on" → RelExtends
- "implements" → RelImplements
- "depends on", "requires" → RelPrerequisiteOf
- "trades off with" → RelTradesOff
```

这项 Prompt 优化是本项目的关键改进。默认 Prompt 仅在 `## See also` 节列出链接，导致关系提取器因检测不到共现的关系动词而无法生成关系边（edges 为 0）。通过强制 LLM 在正文中用强关系词搭配 `[[wikilink]]`，关系连线从 0 条增长至 46 条。

### 3.2 Wiki 链接机制

`[[wikilink]]` 在项目中承担双重角色：

- **Obsidian 标记**：所有 `[[wikilink]]` 在 Obsidian 中自动形成交叉引用边
- **本体关系锚点**：编译器用正则 `\[\[([^\]]+)\]\]` 提取所有 Wiki 链接，同时检测附近的强关系动词，将链接升级为类型化的本体关系边

### 3.3 增量编译

通过 SHA-256 哈希实现。编译器读取 `raw/` 下所有文件的哈希值，与 `.manifest.json` 中记录的旧值比对；哈希不变则跳过，哈希变化则加入编译队列。`.manifest.json` 同时记录编译状态（compiled）、时间戳和概念映射。

## 四、模型设计、选择

### 4.1 配置文件

```yaml
api:
  provider: openai
  api_key: ${SILICONFLOW_API_KEY}
  base_url: https://api.siliconflow.cn/v1

models:
  summarize: Qwen/Qwen2.5-7B-Instruct
  extract:   Qwen/Qwen2.5-14B-Instruct
  write:     Qwen/Qwen2.5-14B-Instruct
  lint:      Qwen/Qwen2.5-7B-Instruct
  query:     Qwen/Qwen2.5-7B-Instruct

embed:
  provider: openai
  model: BAAI/bge-m3
  dimensions: 1024

compiler:
  max_parallel: 4
  debounce_seconds: 2
  summary_max_tokens: 2000
  article_max_tokens: 4000
  auto_commit: true
  auto_lint: true

search:
  hybrid_weight_bm25: 0.7
  hybrid_weight_vector: 0.3
  default_limit: 10
```

### 4.2 模型分流策略

采用**任务驱动的模型分流**，兼顾速度、质量和成本：

| 任务 | 模型 | 理由 |
|---|---|---|
| summarize, lint, query | Qwen2.5-7B-Instruct | 任务简单，7B 即可胜任；查询需毫秒级响应 |
| extract, write | Qwen2.5-14B-Instruct | 需精准 JSON 输出和高质量文章，7B 易产生格式破损 |

### 4.3 API 接入

通过 `base_url: https://api.siliconflow.cn/v1` 对接硅基流动的 OpenAI 兼容 API。API Key 使用环境变量 `${SILICONFLOW_API_KEY}` 引用，避免硬编码。嵌入模型 BAAI/bge-m3 的 `dimensions: 1024` 为必填参数——**这是配置易错点**：不加 `dimensions` 参数，API 可能返回默认维度或报错。

### 6.3 AI 协作排错：实体类型分类修复

这是本项目最具代表性的 AI 协作排错案例。

#### 问题发现

通过 Claude Code MCP 工具查询实体类型分布，发现异常：

| 实体类型 | 数量 |
|---|---|
| concept | 111 |
| **technique** | **0** ❌ |
| claim | 0 |

本该是 technique 的实体（如 supervised-learning、self-attention、RAG、flexbox 等）全被误标为 concept。导致 Web UI 图谱中所有节点都是同一种颜色，缺失 technique 维度。

**[插图6：MCP 工具查询实体类型分布截图]**

#### AI 协作诊断

通过 Claude Code 结合 MCP 工具协作排查：

1. **定位源码**：AI 搜索 sage-wiki Go 代码，发现 `ExtractedConcept` 结构体已定义 `Type` 字段支持三种类型，Go 代码无需修改。
2. **定位根因**：AI 读入 `extract_concepts.txt` Prompt 模板，分析出问题根源——原 Prompt 仅告诉 LLM"可以用三种类型"，但没有给出清晰的分类标准。LLM 面对抽象的 "concept extraction system"，倾向把所有实体都标为 concept。
3. **修复方案**：AI 建议在 Prompt 中添加 Type classification guide，包含：(a) 每种类型的清晰定义和判断标准（用'自问'的方式引导 LLM 判断），(b) 覆盖本项目实际概念的详细示例。

修改前后的核心差异：

```text
# 修改前
- type: "concept", "technique", or "claim"

# 修改后
### Type classification guide
**technique** — 方法、算法、流程。问："这是做某事的方式吗？"
  示例：supervised-learning, self-attention, RAG, flexbox
**concept** — 工具、框架、语言。问："这是一个事物/工具/领域吗？"
  示例：python, react, kubernetes, http
**claim** — 具体断言。问："这是一个具体陈述吗？"
  示例："attention is all you need"
Do NOT default to "concept"
```

#### 验证

修改 Prompt 后重新编译，通过 MCP 工具再次查询，technique 实体从 0 增长至约 23 个。

**[插图7：修复后实体类型分布截图]**

#### 总结

核心教训：**仅告诉 LLM"可以做什么"不够，必须用明确的示例和判断标准告诉它"怎么做"**。该修复完全通过修改 Prompt 模板文件实现，不涉及 Go 代码变更。

### A.2 概念提取 Prompt 模板（核心部分）

```text
For each entity, provide:
- type: "concept", "technique", or "claim"

### Type classification guide

**technique** — A method, algorithm, process, practice, pattern, or architectural
approach that describes HOW to do something.
Ask: "Is this a way of doing something?" If yes → technique.
Examples: supervised-learning, regression, self-attention, pre-training,
RAG, RLHF, mapreduce, flexbox

**concept** — A tool, framework, language, platform, protocol, field of study,
library, service, or abstract idea.
Ask: "Is this a thing/tool/field?" If yes → concept.
Examples: python, react, kubernetes, http, machine-learning, docker

**claim** — A specific assertion, finding, conclusion drawn from a source.
Ask: "Is this a specific statement rather than a named thing?" If yes → claim.
Examples: "attention is all you need", "pretraining improves convergence"

Do NOT default to "concept" — actively evaluate against technique/claim first.
Output ONLY a JSON array of objects. No markdown, no explanation.
```

### A.3 自定义写作 Prompt 模板（核心关系词部分）

```text
CRITICAL INSTRUCTION FOR RELATION EXTRACTION:
You MUST integrate related concepts inline in the article text. When mentioning
a related concept:
1. Wrap it in double brackets [[wikilink]]
2. Explicitly state the relationship using action words:
   - "extends", "builds on" → RelExtends
   - "implements" → RelImplements
   - "optimizes", "improves upon" → RelOptimizes
   - "depends on", "requires" → RelPrerequisiteOf
   - "trades off with" → RelTradesOff
   - "contradicts", "conflicts with" → RelContradicts
Do not just list them in "See also"; explain how they relate in the text.
```
