
# 《Claude Code + Obsidian + sage-wiki：本地LLM知识库探索》项目报告

---

## 一、项目概况

### 1.1 项目背景

sage-wiki 是一个由 Go 语言编写的 LLM 驱动 Graph-RAG 知识库编译器，项目托管于 GitHub（https://github.com/xoai/sage-wiki）。它能够读取原始 Markdown 文档，通过三阶段 LLM 编译管线自动生成结构化 Wiki 知识库。本项目基于 sage-wiki 复现了一套计算机科学多维度知识库系统，以 9 份计算机科学学习路线图作为原始资料，通过编译生成 101 个核心概念文章和 46 条强类型本体关系，构建了一份完整的知识图谱。

### 1.2 技术栈

| 组件 | 技术选型 | 说明 |
|---|---|---|
| 知识库编译器 | sage-wiki (Go) | Graph-RAG 编译管线，三阶段流水线 |
| LLM API | 硅基流动 (SiliconFlow) | 提供 Qwen2.5-7B/14B 等模型 |
| 嵌入模型 | BAAI/bge-m3 | 8192 tokens 上下文窗口 |
| 向量存储 | SQLite + 内置向量索引 | 混合 BM25 + 向量检索 |
| 本地编辑器 | Obsidian | Markdown 知识库可视化 |
| MCP 协议 | Claude Code | 全局 MCP 工具注册 |
| 增量编译 | SHA-256 哈希 + 文件监听 | fsnotify 文件变更检测 |

### 1.3 核心特性

1. **知识库构建**：9 份路线图文档 → 三阶段 LLM 处理 → 结构化 Wiki 输出
2. **Wiki 链接**：Obsidian 兼容的双括号语法 `[[wikilink]]`，支持 Graph-RAG 本体关系提取
3. **增量编译**：SHA-256 哈希比对，仅编译变更的源文件，跳过未变更文件
4. **MCP 接入**：注册到 Claude Code 全局，提供 `wiki_search`、`wiki_ontology_query`、`wiki_capture` 等 MCP 工具
5. **目录设计**：源文档按技术域分类（01-09 编号分区） + 输出目录从 `wiki/` 重命名为 `knowledge-base/`

### 1.4 与 RAG、普通笔记的对比分析

| 维度 | 普通笔记（Obsidian/Notion） | 传统 RAG | 本项目（Graph-RAG Wiki） |
|---|---|---|---|
| 组织方式 | 用户手动建目录、标签 | 向量数据库 + 检索 | 自动提取概念 + 本体关系图谱 |
| 链接语义 | 扁平 `[[链接]]`，无类型 | 无显式链接，仅相似度 | `extends`/`implements`/`depends on` 等强类型 |
| 更新方式 | 用户逐篇编辑 | 重新切片并索引 | 增量编译，仅重新处理变更源 |
| 可解释性 | 依赖人的领域知识 | 黑盒向量相似度 | 图谱可追踪到原始 source，显示推理路径 |
| 知识表达 | 纯文本 Markdown | 碎片化的文本块 | 结构化概念实体 + 显式关系边 + 源文件溯源 |
| AI 协作 | 无内建机制 | 检索 + RAG 生成 | MCP 工具直接查询/写入/本体查询 |

**Graph-RAG 的关键区别**：传统 RAG 是"检索-生成"两段式，本项目先将文档编译为结构化概念图谱（含实体类型、关系边、source 溯源），再进行混合检索。查询结果的上下文不仅来自向量相似文本块，还包括图谱中的直接邻居节点（通过 `extends`、`implements`、`prerequisite_of` 等关系扩展），以及源文档溯源链接。这种"先构建本体知识网，再在知识网上检索"的模式，相比扁平 RAG 显著提升了检索的语义深度和可解释性。

## 三、特征工程

### 3.1 编译管线特征提取流程

sage-wiki 的编译管线是项目的核心"特征工程"环节，负责从原始 Markdown 文档中提取结构化知识。三阶段流水线如下：

**阶段一：Summarize（摘要）**

LLM 读取每个源文档，生成结构化摘要。摘要模板要求输出 Key claims、Methodology、Results、Concepts 四个部分。

使用的 Prompt 模板（`internal/prompts/templates/summarize_article.txt`）：

```text
You are a research assistant creating a structured summary of a source document.

Source file: {{.SourcePath}}
Source type: {{.SourceType}}

Summarize the document with the following structure:

## Key claims
List the main arguments, findings, or assertions.

## Methodology
Describe the approach, methods, or techniques used (if applicable).

## Results
Summarize key results, outcomes, or conclusions.

## Concepts
List key concepts, terms, and ideas introduced or discussed.
Format as a comma-separated list for easy extraction.

Keep the summary under {{.MaxTokens}} tokens. Focus on factual content.
Use precise language. Do not add opinions or commentary.
```

**阶段二：Extract（概念提取）**

LLM 分析所有摘要文档，识别并提取核心概念实体。每个概念实体包括名称（kebab-case）、别名、来源、实体类型。

使用的核心 Prompt 模板（`internal/prompts/templates/extract_concepts.txt`）：

```text
You are an entity extraction system for a knowledge wiki.

Given the following summaries of recently added/modified sources, extract all
notable entities — including concepts, techniques, and claims.

## Known entities (do not duplicate):
{{.ExistingConcepts}}

## New/updated summaries:
{{.Summaries}}

For each entity, provide:
- name: lowercase-hyphenated identifier (e.g., "self-attention")
- aliases: alternative names (e.g., "scaled dot-product attention")
- sources: which source files mention this entity
- type: "concept", "technique", or "claim"

### Type classification guide

Use the following criteria to decide the type. Do NOT default to "concept" —
actively evaluate each entity against the technique and claim criteria first.

**technique** — A method, algorithm, process, practice, pattern, or architectural
approach that describes HOW to do something. Ask: "Is this a way of doing something?"
If yes → technique. Examples: supervised-learning, regression, self-attention,
pre-training, retrieval-augmented-generation, reinforcement-learning-human-feedback,
mapreduce, semantic-html, async-programming, flexbox

**concept** — A tool, framework, language, platform, protocol, field of study,
library, service, or abstract idea. Ask: "Is this a thing/tool/field?"
If yes → concept. Examples: python, react, kubernetes, http, machine-learning,
cloud-computing, cookies, dns, sql, typescript, numpy, jenkins, docker

**claim** — A specific assertion, finding, conclusion, or non-obvious statement
drawn from a source. Ask: "Is this a specific statement rather than a named thing?"
If yes → claim. Examples: "attention is all you need", "pretraining improves convergence"

Merge with known entities when appropriate (detect aliases).
Output ONLY a JSON array of objects. No markdown, no explanation.
```

**阶段三：Write（文章写作）**

LLM 为每个提取的概念实体撰写结构化 Wiki 文章。文章包含 Definition、How it works、Variants、Trade-offs、See also 五个标准章节。

使用的自定义 Prompt 模板（`prompts/write-article.md`）：

```text
You are a wiki author writing a comprehensive article about a concept.

Concept: {{.ConceptName}}
Sources: {{.Sources}}
Related concepts: {{.RelatedList}}

Write a structured wiki article with:

## Definition
Clear, precise definition of the concept.

## How it works
Technical explanation with appropriate depth.

## Variants
Known variants, implementations, or alternatives.

## Trade-offs
Key trade-offs, limitations, or considerations.

## See also
List related concepts as [[wikilinks]]:
{{range .RelatedConcepts}}- [[{{.}}]]
{{end}}

CRITICAL INSTRUCTION FOR RELATION EXTRACTION:
You MUST integrate the related concepts from the Related concepts list inline
in your article's text (specifically in the Definition, How it works, Variants,
or Trade-offs sections). When mentioning a related concept in the text:
1. Wrap it in double brackets [[wikilink]]
2. Explicitly state the relationship in the same sentence or paragraph using
   relationship action words or synonyms such as:
   - "extends", "builds on" (for RelExtends)
   - "implements", "is an implementation of" (for RelImplements)
   - "optimizes", "improves upon" (for RelOptimizes)
   - "depends on", "requires", "prerequisite of" (for RelPrerequisiteOf)
   - "trades off with", "at the cost of" (for RelTradesOff)
   - "contradicts", "conflicts with" (for RelContradicts)
Do not just list them in the "See also" section; actively explain how they
relate in the text using these exact relationship keywords.

Do NOT include YAML frontmatter — it will be added automatically.

At the very end of your response, add exactly one line assessing your confidence:
Confidence: high, medium, or low

Keep under {{.MaxTokens}} tokens. Be precise and factual.
```

**关系词引导演算法规**：通过在 Prompt 中强制要求 LLM 在正文中使用强关系词搭配双括号链接，解决了默认 Prompt 仅在 `## See also` 节列出链接导致关系提取器检测不到共现的关系动词的问题。这使得编译后的 SQLite 数据库中关系连线从 0 条增长至 46 条。

### 3.2 Wiki 链接与本体关系提取

Wiki 链接 `[[wikilink]]` 是 Obsidian 兼容的双括号语法，在本项目中承担双重角色：

**角色 1：Obsidian 图谱标记。** 所有 `[[wikilink]]` 在 Obsidian 中自动形成交叉引用边，提供扁平的非结构化关联网络。

**角色 2：sage-wiki 本体关系锚点。** 编译器扫描概念文章正文，通过正则 `\[\[([^\]]+)\]\]` 提取所有 Wiki 链接。同时检测链接附近的强关系动词（extends、implements、depends on、trades off with、prerequisite of、contradicts、optimizes、derived from 等），将链接升级为类型化的本体关系边，写入 SQLite 数据库的 ontology relations 表。

sage-wiki 支持的关系类型：

| 关系类型 | 含义 | 示例 |
|---|---|---|
| extends | 继承/扩展 | Transformer extends deep-learning |
| implements | 实现 | Kubernetes implements docker |
| optimizes | 优化 | Flash Attention optimizes self-attention |
| prerequisite_of | 前置依赖 | html5 prerequisite_of css3 |
| trades_off | 权衡 | accuracy trades_off interpretability |
| contradicts | 矛盾/冲突 | CAP theorem contradicts ACID |
| derived_from | 来源/派生 | GPT derived_from transformer-architecture |

### 3.3 增量编译的哈希检测

增量编译通过文件 SHA-256 哈希比对实现，`go.sum` 中使用 `crypto/sha256` 标准库进行哈希计算，同时使用 `fsnotify` 库实现文件变更监听。

**哈希检测流程：**

1. 读取 `raw/` 目录下所有源文件的 SHA-256 哈希值
2. 与 `.manifest.json` 中记录的旧哈希值逐一比对
3. 哈希值不变：跳过编译
4. 哈希值变化：标记为待编译，加入编译队列
5. 编译完成后：更新 `.manifest.json` 中的哈希值和时间戳

### 4.1 配置架构

项目通过 `config.yaml` 配置文件统一管理所有 LLM 相关的配置。配置支持环境变量扩展（`${VAR}` 语法），避免 API Key 硬编码。

```yaml
    type: auto

  - path: raw/03-frontend-development
    type: auto
    watch: true
  - path: raw/04-backend-development
    type: auto
    watch: true
  - path: raw/05-fullstack-integration
    type: auto
    watch: true
  - path: raw/06-mobile-development
    type: auto
    watch: true
  - path: raw/07-embedded-systems-and-iot
    type: auto
    watch: true
  - path: raw/08-devops-and-cloud
    type: auto
    watch: true
  - path: raw/09-cybersecurity
    type: auto
    watch: true

output: knowledge-base

# LLM provider configuration
api:
  provider: openai
  api_key: ${SILICONFLOW_API_KEY}
  base_url: https://api.siliconflow.cn/v1

# Model selection per task

  summarize: Qwen/Qwen2.5-7B-Instruct

# Embedding configuration

  provider: openai

  article_max_tokens: 4000
  auto_commit: true

  hybrid_weight_bm25: 0.7
  hybrid_weight_vector: 0.3
  default_limit: 10

serve:
  transport: stdio
  port: 3333
```

### 4.2 API 接入：硅基流动 (SiliconFlow)

硅基流动是国内可用的 LLM API 提供商，支持 OpenAI 兼容的 API 格式。sage-wiki 通过设置 `provider: openai` 和 `base_url: https://api.siliconflow.cn/v1` 实现接入。

**API Key 安全配置**：`api_key: ${SILICONFLOW_API_KEY}` 使用环境变量引用语法，从系统环境变量中读取密钥，避免在配置文件中明文存储：

```powershell
$env:SILICONFLOW_API_KEY = [System.Environment]::GetEnvironmentVariable(
    "SILICONFLOW_API_KEY", "User"
)
```

### 4.3 模型分流策略 (Model Split)

项目采用**任务驱动的模型分流策略**，不同编译阶段使用不同体量的模型，兼顾速度、质量与成本：

| 任务类型 | 使用模型 | 参数量 | 选择理由 |
|---|---|---|---|
| summarize（摘要） | Qwen2.5-7B-Instruct | 7B | 摘要任务对模型要求低，7B 模型即足以提取关键信息 |
| extract（概念提取） | Qwen2.5-14B-Instruct | 14B | 需要精准的实体识别和类型分类，14B 模型能更准确地输出结构化 JSON |
| write（文章写作） | Qwen2.5-14B-Instruct | 14B | 写作任务需要逻辑连贯、内容丰富，14B 模型生成质量显著优于 7B |
| lint（质量检查） | Qwen2.5-7B-Instruct | 7B | 质量检查任务较简单，7B 模型足够胜任 |
| query（检索查询） | Qwen2.5-7B-Instruct | 7B | 查询需要快速响应，7B 模型的低延迟是关键优势 |

**分流设计理由**：

1. **成本优化**：高频任务（summarize、lint、query）使用 7B 模型，大幅降低 Token 消耗
2. **质量保障**：关键任务（extract、write）使用 14B 模型，避免 7B 模型输出非标准 JSON 或低质量文章
3. **避免超时**：硅基流动高峰期大模型有限流，14B 模型的专用分配避免了因并发过高导致的 HTTP 超时
4. **响应速度**：查询场景要求毫秒级响应，7B 模型满足即时交互需求

### 4.4 嵌入模型配置

嵌入模型选择 `BAAI/bge-m3`，支持中英双语、8192 个 Token 的上下文窗口：

```yaml
embed:
  provider: openai
  model: BAAI/bge-m3
  dimensions: 1024
```

配置说明：

- **`provider: openai`**：硅基流动提供与 OpenAI 兼容的 embeddings API
- **`model: BAAI/bge-m3`**：BAAI 的 bge-m3 模型在 MTEB 基准上表现优异，支持多语言嵌入
- **`dimensions: 1024`**：指定向量维度。bge-m3 原生为 1024 维，此参数可显式限定输出维度
- **8192 Token 上下文**：比默认的 text-embedding-3-small（8191）稍大，完全覆盖最长路线图文档

> **注意**：如果不加 `dimensions` 参数，硅基流动 API 可能返回默认维度或报错。该参数是配置 embed 时的易错点。

### 4.5 Wiki 编译流程使用 model 配置

在编译流程的源码中，各种模型配置被读取并应用于不同的任务阶段。在 sage-wiki 源码中，`internal/config/config.go` 定义了 `ModelsConfig` 结构体用于存储模型选择：

```go
type ModelsConfig struct {
    Summarize string `yaml:"summarize"`
    Extract   string `yaml:"extract"`
    Write     string `yaml:"write"`
    Lint      string `yaml:"lint"`
    Query     string `yaml:"query"`
}
```

在 `cmd/sage-wiki/main.go` 的 `runCompile` 函数中，编译流程调度了所有的任务阶段。同时，在 `internal/compiler/fullpipeline.go` 中，`runFullPipeline` 函数负责执行三阶段流水线。该函数读取 `cfg.Models.Summarize`、`cfg.Models.Extract`、`cfg.Models.Write` 等字段，并为每个阶段配置对应的 LLM 模型和客户端参数。

**编译阶段模型分配（fullpipeline.go 逻辑）：**

- **Pass 1 (Summarize)**：使用 `cfg.Models.Summarize`（Qwen2.5-7B-Instruct），设置 `max_tokens` 为 `cfg.Compiler.SummaryMaxTokens`（2000）
- **Pass 2 (Extract)**：使用 `cfg.Models.Extract`（Qwen2.5-14B-Instruct），由 `extract_concepts.txt` Prompt 模板驱动
- **Pass 3 (Write)**：使用 `cfg.Models.Write`（Qwen2.5-14B-Instruct），由自定义的 `write-article.md` Prompt 模板驱动

| classification（分类） | technique | concept |
| self-attention（自注意力） | technique | concept |
| flexbox | technique | concept |
| pre-training（预训练） | technique | concept |
| retrieval-augmented-generation（RAG） | technique | concept |

说明：该截图展示了通过 Claude Code MCP 调用 `wiki_search` 和 `wiki_ontology_query` 查询实体类型分布的结果，证明 technique 类型数量为 0。

#### 6.3.2 与 Claude Code AI 协作诊断

通过 Claude Code 和 MCP 工具进行协作排查：

**Step 1：AI 定位 Go 源码中的类型定义**

AI 搜索 sage-wiki Go 源代码，发现：

- `internal/compiler/concepts.go` 中的 `ExtractedConcept` 结构体已定义 `Type string` 字段支持 concept/technique/claim
- `internal/prompts/templates/extract_concepts.txt` 中的 Prompt 模板也指示了三种类型

结论：Go 代码 **无需修改**，问题出在 Prompt 模板上。

**Step 2：AI 分析 Prompt 问题根源**

AI 读入 `extract_concepts.txt` 内容并分析：原始 Prompt 仅提供了类型名称列表：

```text
- type: "concept", "technique", or "claim"
```

但没有给出清晰的分类标准和具体示例。AI 指出：

- 问题根因：LLM 不清楚何时使用 technique、何时使用 concept。面对抽象的"concept extraction system"，它倾向于把所有实体都打上 concept 标签（因为 concept 是概念提取系统的默认语义）。
- 解决方案：在 Prompt 中添加详细的类型分类指南和具体示例，明确告诉 LLM 三种类型的区别。

**Step 3：AI 协助修改 Prompt 模板**

AI 建议并协助修改了 `extract_concepts.txt`，添加了类型分类指南（Type classification guide）章节，包含：

- 清晰的分类标准（针对每种类型定义一个"自问"问题，帮助 LLM 判断）
- 详细的示例列表（technique 示例、concept 示例、claim 示例，覆盖本项目实际涉及的概念）
- 明确的指令："Do NOT default to 'concept'"

**修改后的 Prompt 模板（extract_concepts.txt 关键部分）：**

```text
For each entity, provide:
- name: lowercase-hyphenated identifier (e.g., "self-attention")
- aliases: alternative names (e.g., "scaled dot-product attention")
- sources: which source files mention this entity
- type: "concept", "technique", or "claim"

### Type classification guide

Use the following criteria to decide the type. Do NOT default to "concept" —
actively evaluate each entity against the technique and claim criteria first.

**technique** — A method, algorithm, process, practice, pattern, or architectural
approach that describes HOW to do something.
Ask: "Is this a way of doing something?" If yes → technique.
Examples: supervised-learning, regression, self-attention, pre-training,
retrieval-augmented-generation, reinforcement-learning-human-feedback,
mapreduce, semantic-html, async-programming, flexbox

**concept** — A tool, framework, language, platform, protocol, field of study,
library, service, or abstract idea.
Ask: "Is this a thing/tool/field?" If yes → concept.
Examples: python, react, kubernetes, http, machine-learning,
cloud-computing, cookies, dns, sql, typescript, numpy, jenkins, docker

**claim** — A specific assertion, finding, conclusion, or non-obvious statement
drawn from a source.
Ask: "Is this a specific statement rather than a named thing?" If yes → claim.
Examples: "attention is all you need", "pretraining improves convergence"

Merge with known entities when appropriate (detect aliases).
Output ONLY a JSON array of objects. No markdown, no explanation.
```

**Step 4：验证修复效果**

修改 Prompt 后重新运行编译（`re-extract` 模式），再次通过 MCP 工具查询实体类型分布：

**[插图6：修复后的实体类型分布截图]**

说明：该截图展示修复后 MCP 查询结果，technique 实体类型不再为 0（约 23 个）。

#### 6.3.3 排错过程总结

本次 AI 协作排错展示了 Prompt 工程在 LLM 驱动的知识库系统中的重要性。核心教训是：**仅仅告诉 LLM"可以做什么"不够，必须用明确的示例和判断标准告诉它"怎么做"**。在 extract_concepts.txt 中添加详细的 Type classification guide 后，LLM 能够准确地区分 concept、technique 和 claim 三种实体类型，生成的关联图中出现了 technique 节点，使得知识图谱的语义表达更加丰富和准确。

### A.5 MCP 服务端初始化代码（internal/mcp/server.go 核心部分）

```go
func NewServer(projectDir string, coordinator ...*compiler.CompileCoordinator) (*Server, error) {
    cfgPath := filepath.Join(projectDir, "config.yaml")
    cfg, err := config.Load(cfgPath)
    if err != nil {
        return nil, fmt.Errorf("mcp: load config: %w", err)
    }

    dbPath := filepath.Join(projectDir, ".sage", "wiki.db")
    db, err := storage.Open(dbPath)
    if err != nil {
        return nil, fmt.Errorf("mcp: open db: %w", err)
    }

    mem := memory.NewStore(db)
    vec := vectors.NewStore(db)
    mergedRels := ontology.MergedRelations(cfg.Ontology.Relations)
    mergedTypes := ontology.MergedEntityTypes(cfg.Ontology.EntityTypes)
    ont := ontology.NewStore(db,
        ontology.ValidRelationNames(mergedRels),
        ontology.ValidEntityTypeNames(mergedTypes))
    searcher := hybrid.NewSearcher(mem, vec)

    // 注册 MCP 工具
    s := &Server{
        projectDir: projectDir,
        db:         db,
        mem:        mem,
        vec:        vec,
        ont:        ont,
        searcher:   searcher,
        cfg:        cfg,
        embedder:   embed.NewFromConfig(cfg),
        language:   cfg.Language,
    }

    // wiki_search: 混合搜索知识库
    s.registerSearchTool()
    // wiki_ontology_query: 本体关系查询
    s.registerOntologyTool()
    // wiki_capture: 对话捕获写入
    s.registerCaptureTool()

    return s, nil
}
```
